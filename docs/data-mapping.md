# Data Mapping – описание исходных данных

## Общая информация
- **Всего файлов:** 5 (`Users.csv`, `Accounts.csv`, `LoyaltyPrograms.csv`, `LoyaltyHistory.csv`, `Offers.csv`)
- **Разделитель:** запятая (`,`)
- **Кодировка:** UTF-8
- **Аномалии:** отмечены в каждом разделе отдельно


## 1. Пользователи (`Users.csv`)

**Назначение:** хранит персональные данные и финансовый сегмент.

| Поле | Тип | Пример | Описание |
|------|-----|--------|----------|
| `id` | int | 1 | Уникальный ID пользователя |
| `email` | str | dmitriy.ivanov29@yandex.ru | Электронная почта |
| `phone_number` | str | +7 994 204-96-79 | Номер телефона |
| `full_name` | str | Иванов Дмитрий Иванович | ФИО |
| `financial_segment` | str | LOW, MEDIUM, HIGH | Сегмент по объёму финансов в экосистеме |

**Проверки:**
- [x] Пропуски отсутствуют
- [x] Типы данных корректны


## 2. Счета (`Accounts.csv`)

**Назначение:** связывает пользователя с программами лояльности и хранит текущий баланс счёта.

| Поле | Тип | Пример | Описание |
|------|-----|--------|----------|
| `account_id` | int | 1 | Уникальный ID счёта |
| `user_id` | int | 1 | Владелец счёта (внешний ключ к `Users.id`) |
| `loyalty_program_id` | int | 1 | ID программы лояльности (внешний ключ к `LoyaltyPrograms.loyalty_program_id`) |
| `current_balance` | float | 82155.16 | Остаток на счёте в рублях |

**Важно:**
- У одного пользователя может быть **несколько счетов**, причём с разными программами лояльности (например, Black и All Airlines одновременно).
- Пропусков нет.


## 3. Программы лояльности (`LoyaltyPrograms.csv`)

**Назначение:** справочник программ лояльности и их валют.

| Поле | Тип | Пример | Описание |
|------|-----|--------|----------|
| `loyalty_program_id` | int | 1 | Уникальный ID программы |
| `loyalty_program_name` | str | All Airlines | Название программы |
| `cashback_currency` | str | miles / rub / bravo-points | Валюта начисления кэшбэка |

**Связи:** используется в `Accounts.loyalty_program_id`.


## 4. История начислений (`LoyaltyHistory.csv`)

**Назначение:** все факты начисления кэшбэка (рубли, мили, баллы) по счетам.

| Поле | Тип | Пример | Описание |
|------|-----|--------|----------|
| `transaction_id` | int | 1 | Уникальный ID начисления |
| `account_id` | int | 1 | Счёт, на который начислено (внешний ключ к `Accounts.account_id`) |
| `cashback_amount` | int | 163 | Сумма в валюте программы |
| `payout_date` | str (date) | 2025-02-23 | Дата начисления (формат `YYYY-MM-DD`) |

**Особенности:**
- Нет привязки к категориям трат – только общая сумма.
- Есть записи с датами в **2026 году** (будущие начисления по графику).
- Все поля заполнены, пропусков нет.

**Пример агрегации:** чтобы получить накопления пользователя по рублям, нужно:
1. Найти все его `account_id` из `Accounts`.
2. Выбрать записи из `LoyaltyHistory` по этим `account_id`.
3. Просуммировать `cashback_amount` отдельно для каждой валюты (валюту определить через `LoyaltyPrograms`).


## 5. Акции партнёров (`Offers.csv`)

**Назначение:** предложения повышенного кэшбэка от внешних партнёров.

| Поле | Тип | Пример | Описание |
|------|-----|--------|----------|
| `partner_id` | int | 1 | Уникальный ID партнёра |
| `partner_name` | str | ЧистоБыстро | Название компании |
| `short_description` | str | Клининговый сервис | Что предлагает |
| `logo_url` | str | https://... | Ссылка на логотип (все используют один временный URL) |
| `brand_color_hex` | str | #2E86DE | Основной цвет бренда (для UI) |
| `cashback_percent` | int | 7 | Размер кэшбэка в процентах |
| `financial_segment` | str | LOW, MEDIUM, HIGH | Для какого сегмента показывать акцию |

**Правила персонализации:**
- HIGH видят все акции (LOW/MEDIUM/HIGH)
- MEDIUM видят MEDIUM + LOW
- LOW видят только LOW

**Замечание:** логотипы временно одинаковые – при реальном внедрении их нужно заменить.


## Итоговые допущения для разработки

1. **Прогноз лояльности** будет строиться на основе `LoyaltyHistory` (без категорий).
2. **Балансы счетов** (`current_balance`) не участвуют в расчёте накопленного кэшбэка, только для сегментации.
3. **Даты** в `payout_date` могут быть будущими – их включаем в историю как есть.
4. Все CSV загружаются в память при старте бэкенда (объём небольшой – 1711 записей в истории).
